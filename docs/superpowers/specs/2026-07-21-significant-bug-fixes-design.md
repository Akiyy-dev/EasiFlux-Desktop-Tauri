# 显著缺陷修复与请求优化设计

日期：2026-07-21
状态：设计已由用户确认并实施，待最终验收

## 背景

项目当前的前端测试、Rust 测试、类型检查、生产构建和现有 CI 级别的 Clippy 检查均可通过，但只读排查发现四项会影响交易风控正确性、环境识别、故障恢复或私有接口请求量的问题：

1. “今日下单次数”仅保存在进程内，不按配置时区自动跨日，也无法跨应用重启保持；额度在 API 请求前直接计入且失败不释放，并发请求还能同时越过上限。
2. 环境检测显示的 Base URL 与实际探测目标可能不一致；一次前端刷新还会重复触发检测。
3. 交易对首次加载失败后会永久保留已完成的 Promise，当前进程不再重试。
4. 交易中心使用 `v-show` 同时挂载全部页签，隐藏页签也会立即请求成交和平仓等私有数据。

本设计以最小、可测试的边界修复上述问题，不扩展到无关重构。

## 目标

- 将每日下单额度变为按配置时区、跨重启保持、并发安全的严格风控。
- 确保失败的下单请求在正常返回路径上释放预占额度。
- 确保环境状态探测选中的真实 Base URL，且一次刷新只执行一次检测。
- 允许交易对加载从瞬时失败中恢复，同时保留缓存和备用列表。
- 避免隐藏交易页签在首次挂载时发起不必要的私有请求。

## 非目标

- 不修改旧版 `EasiFlux-Desktop` 或 `EasiFlux-SDK`。
- 不修改外部 REST、WebSocket 或 HMAC 协议。
- 不修改现有 `config.toml` schema。
- 不实现多账户并行风控；每日额度保持当前应用级语义。
- 不处理 React 迁移、前端懒加载、格式警告、Rust 死代码警告或大型模块拆分。
- 不自动创建 Git commit。

## 方案选择

采用独立本地风控账本，而不是从服务端订单历史重建，也不把运行状态写入 `config.toml`。

服务端重建需要处理分页、历史可见性和离线失败，无法可靠覆盖请求刚受理但历史尚未可见的窗口。把计数写进 `config.toml` 则会将高频运行状态与低频用户设置耦合。独立账本既能跨重启恢复，又能沿用现有 storage/service 分层。

## 1. 严格每日风控

### 1.1 持久化账本

新增 `RiskUsageStore`，文件位于应用数据目录的 `EasiFlux Desktop/risk_usage.toml`。账本包含：

- `schema_version`
- `trading_day`：按配置时区计算的本地日期
- `timezone`
- `occupied_orders`：当天已经成功或正在请求中的额度
- `updated_at_ms`

写入使用同目录临时文件、刷盘和可恢复替换，避免进程退出时只留下半写文件。为兼容 Windows 不允许直接覆盖现有文件的行为，替换时保留上一份 `.bak`：先完成并刷盘 `.tmp`，再轮换主文件与备份，最后将 `.tmp` 提升为主文件。读取时只信任当前主文件；主文件缺失且存在完整 `.tmp` 时，视为替换中断并恢复临时文件。主文件损坏、不可读，或仅剩无法证明时效性的 `.bak` 时一律 fail-closed，避免回退到旧额度而少计。`.bak` 只用于写入失败时原位回滚，不作为启动时的静默降级来源。该流程不引入新的运行时依赖。

账本文件损坏、不可读或不可写时，应用本身仍可启动；但在风控启用时，新下单必须 fail-closed 并返回明确的风控错误。风控关闭时不读取或更新该账本。

### 1.2 每日状态与预占

`RiskService` 用互斥状态替换现有的单个 `AtomicU32`。内部状态保存当前交易日和已占用额度，时钟使用 `TimeService` 的服务器校准时间；服务器时间不可用时沿用其本地时间回退。交易日边界使用 `AppConfig.trading_day_timezone`。

一次下单按以下顺序执行：

1. 校验数量、订单类型和限价。无法解析、零值或负数数量直接拒绝；限价单的价格必须是正数。
2. 计算当前交易日。如果账本日期或时区与当前值不同，先通过可恢复写入持久化归零后的新交易日。
3. 在同一互斥临界区内检查上限并增加 `occupied_orders`，先持久化再更新内存状态。
4. 返回同时包含交易日和时区的预占凭证，然后才调用下单 API。
5. API 成功时无需再次增加计数，预占额度直接成为正式额度。
6. API 明确失败时，使用预占凭证释放同一交易日、同一时区的一格额度；只有释放后的账本成功落盘才更新内存。连接中断、超时或内部错误无法证明服务端没有受理订单，必须保留额度。

如果应用在请求过程中异常退出，已经持久化的预占额度会保留到当天结束。这可能保守地少放行一次订单，但不会因崩溃突破风控上限。

### 1.3 并发与配置更新

额度检查与预占必须位于同一个进程内互斥临界区，不能使用分离的 `load`/`fetch_add`。当最大额度为 1 时，同一应用进程中的两次并发预占最多只能成功一次。本次不新增操作系统文件锁或单实例约束；若同时启动多个桌面进程，它们之间仍不能共享该互斥保证。

`save_config` 继续通过 `RiskService.update_config` 更新风控配置。下一次预占会使用新的时区和上限；时区改变导致交易日键改变时，按新时区开启新的日账本。

## 2. 环境检测

`get_environment_status` 改为纯读取缓存状态，不再隐式运行调度任务。前端 `refreshEnvironment` 保留“运行一次 `environment` 调度任务，再读取状态”的流程。

`SchedulerService.run_environment` 按以下规则选择探测地址：

1. 优先读取当前账户凭据中的 Base URL。
2. 地址为空时使用 `DEFAULT_BASE_URL`。
3. 使用临时 `ApiClient` 或等价的独立探测客户端设置该 Base URL，并调用公开 server-time 端点。
4. 不修改共享 `ApiClient` 的当前凭据或连接状态。

检测成功或失败都形成完整的 `EnvironmentStatus`，写入缓存并通过现有 `environment:updated` 事件发给前端。网络不可达仍然是状态结果，而不是让应用初始化失败。

## 3. 交易对失败重试

`instrumentsFetchPromise` 只表示当前正在执行的请求，并在 `finally` 中清空。新增成功标记，只有解析出非空交易对列表时才认为本次进程已经完成加载。

- 同一时间的多个调用共享一个进行中的 Promise。
- 成功后后续调用直接返回，不重复请求。
- 请求失败或返回空列表时不设置成功标记，下一次调用允许重试。
- 请求期间优先展示 localStorage 缓存；没有缓存时展示配置中的备用列表。
- 后续成功会覆盖缓存、清除错误状态并更新 symbols。

## 4. 交易中心页签请求

保留 `v-show`，并移除父容器随活动页签变化的动态 `key`，从而真正保留表格筛选、选择和滚动状态，不改成切换即销毁的 `v-if`。

调整刷新触发条件：

- `useOrderCenterRefresh` 仅在组件挂载且页签当前激活时刷新。
- 页签从非激活切换为激活时刷新一次。
- `TradeFillsTab` 和 `ClosedPnlTab` 使用同样的激活判断。
- 隐藏页签首次挂载不请求；用户手动刷新仍然立即执行。
- Rust Scheduler 现有的私有面板同步保持不变。

## 5. 错误处理

- 风控账本预占失败：不调用交易 API，返回 `AppError::Risk`。
- 下单 API 明确拒绝且额度释放成功：返回原始 API 错误。
- 下单 API 返回连接中断、超时或内部错误：结果不确定，保留额度并返回原始错误。
- 下单 API 失败且额度释放持久化也失败：保留额度，返回同时包含原始失败与风控账本警告的组合错误；独立日志事件属于后续可观测性增强，不在本次范围内。
- 环境不可达：写入 `reachable = false` 和用户可读错误，继续运行应用。
- 交易对加载失败：保留缓存/备用数据和错误状态，允许后续重试。
- 页签刷新失败：继续使用现有 `errorService` 和 Rust EventEmitter，不新增错误通道。

## 6. 测试设计

### Rust

- 同一交易日持久化并重新构造服务后恢复计数。
- 配置时区跨日后自动归零。
- 最大额度为 1 时，并发预占最多成功一次。
- 释放失败请求的预占后可再次下单。
- 非法、零值和负数数量被拒绝。
- 账本不可写时预占失败，调用方不会进入 API 请求。
- 环境 getter 只读取缓存。
- 自定义 Base URL 是环境探测客户端的实际地址，且不改变共享 API 客户端。

### 前端

- 交易对第一次失败、第二次成功时调用接口两次并更新列表。
- 并发交易对加载只调用一次接口。
- 缓存/备用列表在失败时仍可展示。
- 一次环境刷新仅调用一次 scheduler task 和一次纯 getter。
- 交易中心初始只刷新激活页签。
- 隐藏的成交和平仓页签不请求，激活后各请求一次。
